## Frontend

### Next.js 15

**Почему:** SSR/SSG из коробки (важно для будущего SEO при превращении в публичный сервис),
App Router — современный стандарт, хорошая TypeScript интеграция.

**Альтернативы:**
- Vite + React SPA — нет SSR, хуже для SEO; отклонён
- Nuxt (Vue) — меньше библиотек для сложных UI-взаимодействий; отклонён
- SvelteKit — меньше экосистема, меньше специалистов для будущего масштабирования; отклонён

### Tailwind CSS 4

**Почему:** Утилитарный подход — быстрая вёрстка без создания CSS-файлов.
Нет конфликтов имён классов. Хорошо работает с компонентным подходом React.
Tailwind 4 использует нативный CSS без PostCSS-конфига.

**Альтернативы:**
- CSS Modules — больше файлов, медленнее; отклонён для MVP
- styled-components / Emotion — runtime-overhead; отклонён
- shadcn/ui — библиотека компонентов на базе Tailwind; будет использована поверх

### @dnd-kit/core

**Почему:** Наиболее современная DnD-библиотека для React.
- Поддержка touch (мобильные устройства)
- Модульная архитектура — берём только нужное
- Accessibility из коробки (ARIA)
- Активная поддержка и сообщество

**Альтернативы:**
- react-dnd — устаревшая архитектура, нет touch-поддержки; отклонён
- Framer Motion drag — только анимации, нет логики drop-зон; отклонён
- HTML5 Drag & Drop API — не работает на mobile; отклонён

### Zustand 5

**Почему:** Минимальный API, нет boilerplate (в отличие от Redux).
Достаточно для глобального состояния этого масштаба приложения.
Хорошо интегрируется с TanStack Query.

**Альтернативы:**
- Redux Toolkit — избыточен для одного разработчика и данного масштаба; отклонён
- React Context + useReducer — плохая производительность при частых обновлениях; отклонён
- Jotai — атомарный подход, чуть сложнее для структурированного состояния; отклонён

### TanStack Query 5

**Почему:** Управляет серверным состоянием — кэширование, инвалидация, retry, loading states.
Избавляет от необходимости хранить серверные данные в Zustand.
Оптимистичные обновления для плавного UX при DnD.

**Паттерн разделения:**
- Zustand → UI-состояние (открыта ли модалка, активный шкаф, drag-состояние)
- TanStack Query → серверные данные (книги, шкафы, полки)

## Backend

### NestJS 11

**Почему:** Даёт модульную структуру из коробки — это прямая реализация модульного монолита.
- Dependency Injection — тестируемость
- Встроенные декораторы для валидации, pipes, guards
- TypeScript как первый класс
- Единый язык с фронтендом — можно шарить типы через packages/types

**Альтернативы:**
- Express.js — слишком мало структуры, всё нужно делать самому; отклонён
- Fastify — быстрее, но нет модульной архитектуры; отклонён
- FastAPI (Python) — разрыв в языке, нет шаринга типов; отклонён
- Go (Gin/Fiber) — высокая производительность, но крутое обучение без явной необходимости; отклонён
- Hono — современный, но молодой; экосистема меньше NestJS; отклонён

### Prisma 6

**Почему:** Type-safe ORM с автогенерацией TypeScript-типов из схемы.
Миграции через `prisma migrate` — удобно и предсказуемо.
Prisma Studio — встроенный GUI для БД в разработке.

**Альтернативы:**
- TypeORM — хуже по типобезопасности (декораторы + типы часто расходятся); отклонён
- Drizzle ORM — более новый, хороший, но меньше интеграций с NestJS; может рассматриваться в будущем
- Knex — query builder, нет ORM-уровня; отклонён

## База данных

### PostgreSQL 17

**Почему:**
1. Реляционные данные — книги, полки, шкафы имеют чёткие связи
2. **JSONB** — гибкое хранение метаданных книг из внешних API (будущий F-11)
3. **PostGIS** — расширение для геопространственных запросов (будущий F-13 "книги поблизости")
4. Зрелость, надёжность, широкое сообщество

**Альтернативы:**
- MongoDB — документная БД хуже для реляционных данных; JOIN-запросы сложнее; отклонён
- SQLite — отлично для MVP, но нет смысла мигрировать при росте; отклонён с первого дня
- MySQL — нет JSONB, нет PostGIS; отклонён

## Инфраструктура

### Docker Compose (локальная разработка)

**Содержимое:** PostgreSQL контейнер.
Frontend и Backend запускаются нативно через pnpm для hot-reload.

### pnpm workspaces (монорепо)

**Почему:** Единый node_modules, быстрая установка, нативный воркспейс без дополнительных инструментов.

**Альтернативы:**
- Turborepo — добавляет кэширование сборки, может быть добавлен позже при необходимости
- Nx — мощный, но избыточен для одного разработчика; отклонён
- Lerna — устаревший подход; отклонён

### Продакшн: Railway / Render

**Почему:** Минимальные DevOps-усилия для одного разработчика.
Managed PostgreSQL, автодеплой из Git, разумная цена на старте.

**При росте:** Миграция на VPS (Hetzner Cloud — цена/качество) с Docker Compose или k3s.
