# Архитектура Visual Library

## Обзор

**Visual Library** — инструмент визуализации домашней книжной библиотеки.
Позволяет создавать виртуальные книжные шкафы, размещать книги на полках и визуально
просматривать расположение книг.

---

## Тип архитектуры: Модульный монолит

### Почему не микросервисы

Проект разрабатывается одним человеком. Микросервисы на этом этапе создают:
- Избыточную сложность деплоя (несколько сервисов, service discovery, межсервисная сеть)
- Высокий overhead на коммуникацию
- Сложность в отладке

### Почему не простой монолит

Простой монолит не имеет чётких границ между доменами, что делает его неподдерживаемым
при росте. При необходимости выделить сервис (например, поиск книг поблизости с геолокацией)
придётся переписывать всё с нуля.

### Почему модульный монолит

- Чёткие границы между доменами (books, bookcases, users, subscriptions) внутри одного приложения
- Простота деплоя как у монолита
- Возможность вынести любой модуль в отдельный сервис без переработки бизнес-логики
- Единый процесс = простая отладка

## Слои системы

### Frontend (Next.js)

```
app/           → роутинг, страницы (Next.js App Router)
components/    → UI-компоненты (разделены на ui/ и доменные)
features/      → feature-модули (логика + компоненты конкретной фичи)
store/         → глобальное состояние (Zustand)
hooks/         → переиспользуемые React-хуки
lib/           → API-клиент, утилиты, константы
```

### Backend (NestJS)

```
modules/       → бизнес-модули (каждый = Controller + Service + Repository)
shared/        → общие декораторы, guards, фильтры ошибок, утилиты
prisma/        → схема БД, миграции, Prisma-клиент
```

### Shared Types (packages/types)

Общие TypeScript-типы, которые используются и на фронте, и на бэке.
Исключает дублирование и рассинхронизацию контрактов.

## API

- **Стиль:** REST
- **Версионирование:** `/api/v1/` — закладывается с первого дня, чтобы будущие breaking changes
  не ломали клиентов
- **Формат:** JSON
- **Коды ответов:** стандартные HTTP (200, 201, 400, 404, 422, 500)

Подробное описание endpoints → [API.md](./API.md)

## Взаимодействие компонентов (Data Flow)

### Пример: пользователь перетаскивает книгу на полку

```
1. UI: @dnd-kit фиксирует drop-событие (onDragEnd в BookcaseDndContext)
2. BookcaseDndContext: определяет источник (spine / unplaced) и цель (shelfId)
3. Вызывается useCreatePlacement или useUpdatePlacement (TanStack Query mutation)
4. API-клиент: POST /api/v1/placements или PATCH /api/v1/placements/:id
5. NestJS PlacementsController → PlacementsService
6. PlacementsService: в транзакции — delete → normalize positions → create
   (избегает UNIQUE(shelfId, position) конфликта при перемещении)
7. Ответ: 201 / 200 с объектом placement
8. onSuccess: TanStack Query инвалидирует кэш шкафа → refetch → UI обновляется
9. sonner toast: «Книга поставлена на полку» / «Книга перемещена»
10. При ошибке: sonner toast с сообщением, UI возвращается в предыдущее состояние
```

## Инфраструктура

### Локальная разработка

Docker Compose запускает PostgreSQL. Frontend и Backend запускаются нативно через pnpm.

### Продакшн (MVP)

Весь стек запускается через Docker Compose на любом VPS (Ubuntu 22.04+):
- **Сервисы:** postgres, migrate (one-shot), api, web, nginx
- **Reverse proxy:** Nginx → `/api/*` → api:3001, `/` → web:3000
- **Запуск:** `docker compose -f docker-compose.prod.yml --env-file .env.prod up -d --build`
- Подробная инструкция → [docs/DEPLOYMENT.md](./DEPLOYMENT.md)

### Будущее (при росте)

- Добавление Redis для кэширования и сессий
- PostGIS для геопространственного поиска
- CDN для обложек книг

## Решения и Trade-offs

| Решение | Альтернатива | Причина выбора |
|---|---|---|
| Модульный монолит | Микросервисы | Скорость разработки, один разработчик |
| PostgreSQL | MongoDB | Реляционные данные + будущий PostGIS |
| Prisma | TypeORM | Лучшая типобезопасность, удобные миграции |
| REST | GraphQL | GraphQL избыточен для этого масштаба |
| Next.js | Vite SPA | SSR/SEO для будущего публичного сервиса |
| @dnd-kit | react-dnd | Модульнее, поддержка touch, активно поддерживается |
| Zustand | Redux | Проще для одного разработчика, достаточно для масштаба |
| CSS/DOM визуализация | Canvas (Konva.js) | Нативный DnD, доступность, проще анимировать |

## Ограничения текущей архитектуры

1. **Нет real-time** — при добавлении совместного доступа к библиотекам потребуется WebSocket
   (можно добавить через NestJS Gateway без смены архитектуры)
2. **Нет CDN** — обложки книг хранятся как URL, не как файлы. При добавлении загрузки
   изображений потребуется S3-совместимое хранилище
3. **Нет кэширования** — при росте нагрузки потребуется Redis
